## AI 코딩 에이전트 개발 지침 (GEMINI.md)

### 1. 기본 설정

* 모든 응답은 한국어로 한다.
* 프로젝트 정보를 `memory.md`에 지속적으로 기록하며 진행한다.

### 2. 검증과 확인의 원칙 (검증 우선)

* 가정과 추측은 절대 허용되지 않는다. 리소스를 추가로 들이더라도 확인된 사실만을 바탕으로 개발한다.
* 문서나 주석이 아닌 코드를 직접 읽는다. 주석은 최신 상태를 반영하지 않을 수 있으므로 신뢰하지 않는다.
* 모든 코드는 tool을 이용해 직접 읽어야 한다. 양이 아무리 방대하더라도 뛰어넘지 않는다.
* 어떤 코드를 import할 때도 존재와 내용을 직접 확인한다. 추측으로 import하지 않는다.
* import 전에 해당 모듈의 존재와 내용을 반드시 확인한다.
* .env .local 등 보안 파일은 AI 접근이 제한되어 있음을 인지하고, 이러한 파일의 존재나 내용을 가정하지 않는다. API KEY사용이 필요한경우 dotenv를 사용한다.

### 3. 기존 구조와 패턴 준수

* 코드 작성 전 반드시 프로젝트의 기존 CRUD 패턴, 데이터 흐름, 폴더 구조를 완전히 분석한다.
* 분석 완료 전에는 어떠한 코드도 작성하지 않는다.
* 새로운 코드는 기존 프로젝트의 아키텍처, 네이밍 컨벤션, 코드 스타일과 완벽히 일치해야 한다.
* 기존 패턴을 임의로 변경하거나 무시하지 않는다.

### 4. Single Source of Truth 원칙

* 모든 타입, 상수, 설정값은 단 한 곳에만 정의한다.
* 코드 작성 전 기존에 정의된 타입, 상수, 유틸리티 함수가 있는지 반드시 확인한다.
* 중복 정의를 절대 하지 않으며, 기존 정의를 import하여 사용한다.
* 문자열 상태값, 숫자, URL, API 엔드포인트, 에러 메시지, 정규식, 타임아웃 값 등 모든 리터럴 값은 상수로 분리한다.
* 같은 값이 2회 이상 사용되거나, 비즈니스 로직과 관련되거나, 향후 변경 가능성이 있는 모든 값은 반드시 상수화한다.
* 2회 이상 사용되는 값은 상수로 분리한다.

### 5. 단일 책임과 모듈화

* 함수는 단 하나의 기능만 수행하고, 파일은 단 하나의 책임만 갖는다.
* 재사용 가능한 코드는 별도 파일로 분리한다.
* 2회 이상 재사용될 가능성이 있는 모든 함수와 컴포넌트는 Shared 폴더로 분리한다.
* 개발 완료 후 반드시 재사용 가능한 코드를 식별하여 Shared로 이동한다.
* 모든 로직을 한 파일에 작성하지 않고 책임에 따라 적절히 분리한다.

### 6. 에러 처리 및 품질 기준 (에러 즉시 해결)

* 모든 에러는 처음 발견 시 즉시 해결한다. 나중으로 미루지 않는다.
* Linter 규칙을 처음부터 100% 준수한다.
* 유저의 80%는 예상치 못한 행동을 한다는 전제로 모든 예외 상황을 처리한다.
* Happy Path만 고려한 코드는 불완전한 코드로 간주한다.
* console.log로 에러를 무시하거나, any 타입을 사용하거나, try-catch로 에러를 묵살하는 등의 임시방편을 사용하지 않는다.
* 모든 에러에 대해 적절한 에러 메시지와 복구 전략을 구현한다.
* 근본 원인을 파악하고 해결한다.

### 7. 정석 구현 원칙

* 모든 코드는 tool을 이용해 직접 작성한다.
* 문제해결과 구현은 항상 정석적인 방법으로만 수행한다.
* 현상만 해결하거나 보이기만 하는 임시방편을 절대 사용하지 않는다.
* 근본 원인을 해결하는 올바른 솔루션만을 구현한다.

### 8. 명령 준수와 집중

* 지시받은 내용만 수행하고 질문받은 내용에만 답한다.
* 요청받지 않은 제안이나 개선사항을 먼저 제시하지 않는다.
* 현재 수행 중인 목표에만 집중하며, 관련 없는 코드를 개선하지 않는다.
* 일반적인 개선사항이나 리팩토링은 명시적 요청이 있을 때만 수행한다.

### 9. 최소 문서화 원칙

* Code를 유일한 Source of Truth로 유지한다.
* 주석과 문서는 반드시 필요한 경우에만 최소한으로 추가한다.
* 자명한 코드를 작성하여 주석의 필요성을 원천적으로 줄인다.

### RAG 시스템 및 에이전트 특화 규칙

* **VectorDB 재사용**: 기존 `faiss_index/` 폴더가 존재하면 새로 생성하지 않고 기존 인덱스를 로드한다.
* **수준별 필터링**: 모든 RAG 검색 시 사용자의 숙련도(Level 1~5) 정보를 전달하여 YAML 메타데이터 기반의 필터링을 수행한다.
* **멀티모달 분석(Vision)**: 악보 이미지 업로드 시 Gemini의 Vision 기능을 활용하여 코드를 텍스트로 추출하고, 이를 분석 도구로 연결한다.
* **LCEL 사용 권장**: LangChain의 LCEL 패턴을 사용하여 분석 파이프라인을 구성한다.
* **API Key 관리**: API Key는 반드시 `.env` 파일과 `python-dotenv` 라이브러리를 통해 관리한다.
* **LLM**: `gemini-3.5-flash`를 기본으로 사용하며, 분석 정확도를 위해 `temperature`는 낮은 값(0.1)을 권장한다.
* **에이전트**: 사용자의 코드 진행 입력을 받으면 [기능 분석 -> 기법 탐지 -> 연주 가이드]의 논리적 흐름으로 도구를 호출한다.

## 필수 검증 체크리스트

* 기존 프로젝트 패턴을 완전히 분석했는가
* 모든 값이 적절히 상수화되었는가
* 중복 정의가 존재하지 않는가
* 모든 예외 상황이 처리되었는가
* Linter 경고가 하나도 없는가
* 임시방편 없이 정석으로 구현했는가
